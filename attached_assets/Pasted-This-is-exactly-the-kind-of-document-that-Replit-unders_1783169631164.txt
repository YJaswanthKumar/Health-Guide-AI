# NutriWise AI - Complete Project Architecture & AI Agent Specification

## 1. Project Overview

### Project Name

**NutriWise AI – Intelligent Health, Nutrition & Wellness Platform**

### Vision

NutriWise AI is not simply an AI chatbot. It is an intelligent personal healthcare companion that assists users throughout their health journey.

The system combines:

* Health assessment
* Medical report analysis
* Personalized nutrition
* Care planning
* Emergency assistance

into one integrated AI ecosystem.

Unlike traditional chatbots that simply answer questions, NutriWise AI continuously understands the user's medical condition, lifestyle, nutrition, medications, recovery progress, daily habits, and historical data to provide personalized recommendations.

The website serves as the user interface and workflow orchestrator.

CrewAI serves as the AI reasoning engine.

---

# Core Philosophy

The website should NEVER contain medical reasoning.

Instead:

Frontend

↓

Backend

↓

↓

CrewAI Agent

↓

↓

Agents never communicate directly.

They communicate through structured outputs handled by the backend.

---

# Why Multiple AI Agents?

Instead of one huge prompt, responsibilities are separated.

Benefits:

* Easier maintenance
* Independent testing
* Better accuracy
* Easier upgrades
* Lower prompt complexity
* Easier debugging
* Future scalability

---

# Agent Communication

Every agent follows the same rule.

INPUT

↓

Reasoning

↓

OUTPUT JSON

↓

Backend

↓

Next Agent

No agent should directly call another agent.

The backend orchestrates everything.

---

# Standard Input Format

Every agent receives JSON.

Example

```json
{
   "user_profile":{},
   "conversation_history":[],
   "daily_logs":[],
   "current_plan":{},
   "medical_conditions":[]
}
```

No plain text communication between agents.

---

# Standard Output Format

Every agent returns structured JSON.

Example

```json
{
   "status":"success",

   "summary":"...",

   "recommendations":[],

   "warnings":[],

   "next_agents":[],

   "backend_actions":[]

}
```

This makes every agent interchangeable.

# Backend Responsibilities

The backend is the brain of the application.

It should:

* Authenticate users
* Store data
* Route requests to CrewAI
* Execute backend actions (email, SMS, DB writes)
* Merge responses
* Cache results
* Stream AI responses
* Handle retries and errors

It **must not** perform medical reasoning.

# Agent Workflow Examples

### Medical Report Upload

1. User uploads PDF.
2. Backend stores the file.
3. Agent 1 extracts and analyzes the report.
4. Backend saves structured data.
5. If nutrition changes are needed, Agent 5 is called.
6. If the care plan changes, Agent 3 is called.
7. If a critical condition is detected, Agent 4 is called.
8. Dashboard updates automatically.

### Symptom Chat

1. User describes symptoms.
2. Agent 2 asks follow-up questions until enough information is gathered.
3. Agent 2 produces an assessment.
4. Backend updates the profile only after user confirmation.
5. Agent 3 updates the care plan.
6. Agent 5 generates personalized nutrition advice.
7. Agent 4 is triggered only if the severity is high.

---

# Design Principles

* **Single Responsibility:** Each agent has one clear purpose.
* **Backend Orchestration:** Agents never call each other directly.
* **Structured Communication:** All interactions use JSON.
* **Human-in-the-Loop:** Profile updates and irreversible actions require confirmation.
* **Evidence-Based:** Nutrition advice is grounded in retrieved documents (RAG), not model memory.
* **Safety First:** Emergency and medical recommendations are cautious, non-diagnostic, and designed to support—not replace—professional medical care.
